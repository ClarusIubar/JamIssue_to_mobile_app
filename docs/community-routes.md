# Community Routes

사용자가 직접 찍은 스탬프 동선을 공개 경로로 발행하고, 다른 사용자는 그 경로를 좋아요순과 최신순으로 탐색하는 기능이다.

## PRD 반영 문장

`사용자가 실제 방문한 스탬프 기반 동선을 공개 경로로 발행하고, 다른 사용자는 좋아요순/최신순으로 그 경로를 탐색할 수 있어야 한다.`

## 목표
- 운영자가 임의 조합한 코스만 보여주지 않는다.
- 실제 방문 이력을 가진 사용자 경로를 추천 레이어로 끌어올린다.
- 좋아요가 많은 경로가 먼저 노출되도록 해서 사용자 생태계를 만든다.

## 현재 규칙
- 경로 공개는 로그인 사용자만 가능하다.
- 경로에는 최소 2곳, 최대 6곳까지 넣을 수 있다.
- 내가 실제로 스탬프를 찍은 장소만 경로에 넣을 수 있다.
- 공개 경로 목록 정렬은 두 가지다.
  - `popular`: 좋아요 수 내림차순, 같으면 최신순
  - `latest`: 생성 시각 내림차순
- 좋아요는 로그인 사용자만 가능하다.
- 내가 만든 경로에는 좋아요를 누를 수 없다.

## 좋아요 스키마 원칙
- `피드 좋아요`와 `경로 좋아요`는 별도 테이블로 유지한다.
- 현재 기준 테이블은 `feed_like`, `user_route_like` 두 개다.
- 이유는 대상 엔터티가 다르고, FK 무결성과 중복 제어를 SQL 레벨에서 바로 보장하기 쉽기 때문이다.
- `target_type`, `target_id` 하나로 합치는 범용 `like` 테이블은 유연해 보여도 FK를 강하게 걸기 어렵고 운영 실수도 늘어난다.

## user_id 결합도에 대한 판단
- 현재 구현은 내부 `user.user_id`를 고정 계정 ID로 쓰고, 외부 로그인 식별자는 `user_identity(provider, provider_user_id)`로 분리했다.
- 그래서 네이버와 카카오를 오가더라도 내부 `user_id`가 유지되면 경로, 후기, 좋아요, 스탬프가 끊기지 않는다.
- 기본 로그인에서는 같은 이메일이어도 자동으로 계정을 합치지 않는다.
- 계정 병합은 이미 로그인한 사용자가 명시적으로 계정 연결을 할 때만 허용한다.
- 이 방식이 공급자별 이메일 정책 차이로 생길 수 있는 잘못된 병합을 막는 데 더 안전하다.

## 데이터 모델
- `user_route`
  - 경로 메타 정보
  - 작성자, 제목, 소개, 무드, 공개 여부, 좋아요 수
- `user_route_place`
  - 경로에 포함된 장소와 순서
- `user_route_like`
  - 사용자별 경로 좋아요 이력
- `feed_like`
  - 사용자별 장소 후기 좋아요 이력

## API
- `GET /api/community-routes?sort=popular|latest`
  - 공개 경로 목록 조회
- `POST /api/community-routes`
  - 내 스탬프 기반 경로 생성
- `POST /api/community-routes/{route_id}/like`
  - 경로 좋아요 토글
- `GET /api/my/routes`
  - 내가 만든 경로 목록 조회

## 프론트 연결
- `코스` 탭
  - 공개 경로 목록 노출
  - 좋아요순 / 최신순 전환
  - 경로 내 장소 칩 클릭 시 탐색 탭 상세로 이동
- `마이` 탭
  - 스탬프 기반 경로 작성 폼
  - 내가 만든 경로 목록 확인
